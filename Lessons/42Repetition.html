Learning .

QBasic can get data from several sources including the user and external files.  Sometimes, however, the program will use static data inside the program itself.  This might be a string of notes to play a tune.  It might be color sequences.  It could be pretty much anything the programmer wants to store "permanently."  (I say permanently because the program actually has to be modified to change the data if it's stored this way.)

Storing data in the program is done using DATA statements.  These are usually at or near the end of the program.  The data is accessed by using the READ command which assigns the values to variables during the execution of a program.

David I. Schneider's "QBASIC with an Introduction to Visual Basic 5.0" on page 59, has a good discussion of this topic.

DATA statements store values to be assigned to variables by READ statements during the execution of a program.  If a DATA statement contains more than one item of data, the items are separated by commas.  Examples of DATA statements are

DATA "Mike Jones", 7.35, 35

DATA "John Smith", 6.75, 33

These six items of data could have been placed in one DATA statement or in six different DATA statements.  The items of data will be assigned to variables one at a time in the order they appear in the DATA statements.  That is, "Mike Jones" will be the first value assigned to a variable.  After all the items from the first DATA statement have been assigned to variables, subsequent requests for values will be read from the next DATA statement.  Although DATA statements can appear anywhere in a program, they are usually placed at the end, just before the END statement.  This makes them easy to find and modify.  If var is a variable, then the statement

READ var

causes the program to look in the DATA statements for the next available item of data and assign it to the variable var.

This is an example of a program which reads and prints some info from DATA statements.

REM plong
REM Data Statement example without loop
REM 6-5-2005

DIM username AS STRING, score AS SINGLE, age AS INTEGER

CLS

REM This is a header
PRINT "User", " Score", " Age"

READ username, score, age
PRINT username, score, age

READ username
READ score
READ age
PRINT username, score, age

DATA "Mike Jones", 7.35, 35
DATA "John Smith", 6.75, 33

END

Notice that this contains several READ statements.  It could have been done with a minimum of two READS, but the latter set is for example.

A problem like this is easily solved without a loop, but it would become cumbersome if there were several thousand, or even a hundred sets of data.  A loop will make this program much improved.

Since we know how much data is in the program we can use a for next loop.

REM plong
REM Data Statement example - For-Next loop
REM 6-5-2005

DIM username AS STRING, score AS SINGLE, age AS INTEGER
DIM i AS INTEGER

CLS

REM This is a header
PRINT "User", " Score", " Age"

FOR i = 1 TO 2
     READ username, score, age
     PRINT username, score, age
NEXT i

DATA "Mike Jones", 7.35, 35
DATA "John Smith", 6.75, 33

END

This will be much more manageable.  Play with it to see what happens if you try to read more data than is available.

If we had an undetermined amount of data we wouldn't want to use a For-Next loop.  A better choice would be the Do-While or the Do-Until.

REM plong
REM Data Statement example without loop
REM 6-5-2005

DIM username AS STRING, score AS SINGLE, age AS INTEGER
DIM i AS INTEGER

CLS

REM This is a header
PRINT "User", " Score", " Age"

READ username, score, age

DO
     PRINT username, score, age
     READ username, score, age
LOOP UNTIL username = "EOD"

DATA "Mike Jones", 7.35, 35
DATA "John Smith", 6.75, 33
DATA "EOD", -1, -1

END

Notice that the Do loop requires a bit of pre-loop-processing.  It also requires a trailer value or sentinel.  This value must be something which absolutely would not possibly be part of the data.  In this case I've used "EOD" for the string sentinel and -1 for the numeric sentinel.  This is necessary to inform the loop to stop.

The RESTORE command is another important part of processing DATA, but this will be left for another time.
